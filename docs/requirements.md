# Story Point Porker - 要件定義書

## 1. プロジェクト概要

### 1.1 アプリケーション名
**Story Point Porker** - プランニングポーカーアプリ

### 1.2 目的
スクラムチームがストーリーポイントの見積もりを行うための、リアルタイムマルチプレイヤー型プランニングポーカーWebアプリケーション。

### 1.3 想定ユーザー
- スクラムチーム（5〜10人規模）
- スクラムマスター / ファシリテーター（ホスト役）

---

## 2. 技術スタック

| 技術領域 | 採用技術 |
|----------|----------|
| フロントエンド | SvelteKit |
| リアルタイム通信 | Firebase Realtime Database |
| 認証 | なし（名前入力のみ） |
| ホスティング | 未定（Firebase Hosting / Vercel等） |

---

## 3. 機能要件

### 3.1 ルーム管理

#### 3.1.1 ルーム作成
- ホストがルームを作成すると、一意のルームIDを含むURLが生成される
- 例: `https://example.com/room/{roomId}`
- ルーム作成者は自動的にホスト権限を持つ

#### 3.1.2 ルーム参加
- URLにアクセスした参加者は、表示名を入力して入室する
- アカウント登録・ログインは不要
- 参加者一覧がリアルタイムに更新される

#### 3.1.3 ルーム退出
- ブラウザを閉じる / 退出ボタンで退室
- 退室した参加者は参加者一覧から削除される

### 3.2 投票（プランニングポーカー）

#### 3.2.1 カード数列
フィボナッチ風の数列を使用:

```
1, 2, 3, 5, 8, 13, 21
```

#### 3.2.2 カード選択
- 各参加者はカード一覧から1枚を選択する
- 選択後、他の参加者には「投票済み」であることのみ表示される（値は非公開）
- 投票後にカードを変更することも可能（公開前まで）

#### 3.2.3 一斉公開
- **ホストのみ**が「カードを公開する」ボタンを押せる
- 全参加者のカードが同時に公開される
- 公開時に派手なアニメーション演出を行う

#### 3.2.4 リセット
- **ホストのみ**が「リセット」ボタンを押せる
- 全員のカード選択がクリアされ、次の投票に備える

### 3.3 議題（イシュー）管理

#### 3.3.1 議題登録
- ホストが複数の議題をテキストで登録できる
- 議題にはタイトルと任意の説明文を設定可能

#### 3.3.2 議題進行
- **ホストのみ**が「次の議題へ」ボタンで進行を制御
- 現在の議題が画面上部に表示される
- 投票結果は議題に紐づけて保存される

#### 3.3.3 議題一覧
- 登録済みの議題一覧が表示される
- 各議題のステータス（未投票 / 投票済み / 確定）がわかる
- 投票済みの議題には確定したポイントが表示される

### 3.4 統計表示

投票結果の公開後に以下の統計情報を表示する:

| 統計項目 | 説明 |
|----------|------|
| 平均値 | 全投票の平均 |
| 中央値 | 全投票の中央値 |
| 分布 | 各カードに何人が投票したかの棒グラフ |
| 投票数 | 投票した人数 |

### 3.5 ホスト権限

ホスト（ルーム作成者）のみが実行できる操作:

- カードの一斉公開
- 投票のリセット
- 次の議題への進行
- 議題の登録・編集・削除
- ポイントの確定

一般参加者ができる操作:

- カードの選択・変更
- 参加者一覧の閲覧
- 投票結果・統計の閲覧

### 3.6 テーマ切替

- ネオンテーマ（デフォルト）と Windows XP Luna テーマを切替可能
- テーマ設定は `localStorage` に永続化
- XP テーマ時にはクリッピー風のイルカアシスタントが表示される

### 3.7 自動クリーンアップ

- ホストがブラウザを閉じた場合、Firebase `onDisconnect` でルーム全体を自動削除
- プレイヤーがブラウザを閉じた場合、Firebase `onDisconnect` で該当プレイヤーデータを自動削除

### 3.8 URL自動リンク

- 議題タイトル中の URL（`https://...`）を自動検出し、クリック可能なリンクとして表示

### 3.9 投票状況インジケータ

- 投票済み/未投票の人数をリアルタイムで表示（例: `3/5 選択済み`）
- 全員投票完了時に視覚的に強調表示

## 4. 画面構成

### 4.1 トップページ (`/`)
- アプリタイトル・ロゴ
- 「ルームを作成する」ボタン
- ルームIDまたはURL入力による参加フォーム

### 4.2 入室画面 (`/room/{roomId}` - 未入室時)
- 表示名入力フォーム
- 「参加する」ボタン

### 4.3 ルーム画面 (`/room/{roomId}` - 入室後)
メイン画面。以下の要素で構成:

- **ヘッダー**: ルーム名、現在の議題
- **テーブルエリア**: 参加者のカードを円形/楕円形に配置（ポーカーテーブル風）
- **カード選択エリア**: 画面下部に手札として数列カードを表示
- **サイドバー**: 議題一覧（ホストは編集可能）
- **統計パネル**: 公開後に表示される結果パネル
- **コントロールバー**: ホスト用の操作ボタン（公開・リセット・次の議題）

## 5. UIデザイン要件

### 5.1 テイスト
**パチンコ風 - 派手でカラフル**（ネオンテーマ） / **Windows XP Luna**（XPテーマ）

### 5.2 カラーパレット
- ネオンカラーを基調（ピンク、シアン、イエロー、グリーン）
- 背景はダーク系（ネオンが映えるように）
- グラデーションやグロウエフェクトを多用

### 5.3 アニメーション
- カード選択時: カードが光るエフェクト
- カード公開時: 派手なフリップアニメーション + パーティクルエフェクト
- 全員一致時: 特別な大当たり演出
- 参加者入室時: 入室アニメーション

### 5.4 フォント
- 太めのゴシック体 / インパクトのあるフォント
- 数字は大きく視認性を重視

## 6. データモデル（Firebase Realtime Database）

```
rooms/
  {roomId}/
    id: string
    createdAt: timestamp
    hostId: string
    currentIssueIndex: number
    status: "voting" | "revealed"
    issues/
      {issueId}/
        id: string
        title: string
        result: string | null
    players/
      {playerId}/
        id: string
        name: string
        joinedAt: timestamp
        isHost: boolean
        vote: string | null
```

## 7. 非機能要件

### 7.1 パフォーマンス
- リアルタイム同期の遅延: 500ms以内
- 初回読み込み: 3秒以内

### 7.2 ブラウザサポート
- Chrome, Firefox, Safari, Edge の最新版

### 7.3 レスポンシブ
- PC（メイン対応）
- タブレット（基本対応）
- スマートフォン（最低限の対応）

### 7.4 セキュリティ
- Firebase Security Rules でルーム単位のアクセス制御
- 他ルームのデータにはアクセス不可
